# Feature: Program Advancement & Season Lifecycle

> Advance the user's program day, detect season completion at day 90, trigger automatic season rollover, and celebrate milestones.

Feature: Program Advancement & Season Lifecycle. Actor: System, User (Member). Phase: MVP.
Implementation: `src/services/program/programAdvancement.ts`, `supabase/functions/season-rollover-handler/index.ts`, `app/(app)/season-complete.tsx`, `src/hooks/useProgramAdvancement.ts`
Purpose: Advance the user's program day, detect season completion at day 90, trigger automatic season rollover, and celebrate milestones.

---

## 1.1 Program Day Advancement (Daily Check on App Open, Recovery for Lapsed Users)

Program day advancement happens via `advanceProgramDay(userId)` in `programAdvancement.ts`, called by `useProgramAdvancement.checkAdvancement()` when the user opens the app. The function:

1. Reads the user's current `program_day`, `season`, and `plan_start_date` from the `users` table.
2. Reads the `last_active_date` from the `progress` table.
3. If `last_active_date` equals today, returns immediately (already advanced today).
4. Calculates the gap between `last_active_date` and today — if gap > 1, flags `wasInactive = true` (used for recovery messaging but does NOT affect program day — the user continues from their last day without penalty).
5. Increments `program_day` by 1, capped at 90.
6. Updates `users.program_day` with an optimistic-concurrency guard: `UPDATE ... WHERE program_day = oldValue` — prevents double-advancement from concurrent opens.
7. Updates `progress.last_active_date` to today and `updated_at` to now.
8. Returns `ProgramDayResult` with `{ programDay, season, daysAdvanced, seasonComplete, wasInactive }`.

The function is idempotent: if called twice on the same day, the `last_active_date === today` check returns early. The optimistic-concurrency guard on the update prevents race conditions if the app is opened on multiple devices simultaneously.

- Reads `program_day`, `season`, `plan_start_date` from `users`
- Reads `last_active_date` from `progress`
- Early return if `last_active_date === today` (already advanced)
- Gap detection: `daysBetween(lastActive, today) > 1` → `wasInactive = true` (no penalty, just flag)
- Program day increments by 1, capped at 90 (`Math.min(program_day + 1, 90)`)
- Optimistic-concurrency guard: `UPDATE users SET program_day = newValue WHERE id = userId AND program_day = oldValue`
- Updates `progress.last_active_date` and `updated_at`
- Returns `{ programDay, season, daysAdvanced, seasonComplete, wasInactive }`

## 1.2 Season Complete Detection (Day 90)

Season completion is detected when `advanceProgramDay()` advances the program day to 90 for the first time (i.e., `newProgramDay === 90 && user.program_day < 90`). The function returns `seasonComplete: true` in its result. The `useProgramAdvancement` hook sets `isSeasonComplete = true` and stores the full `ProgramDayResult` in `seasonCompleteData` state. The app's root layout or home screen checks `isSeasonComplete` and, when true, navigates to the season-complete screen for the celebration flow before allowing normal app interaction. The season-complete flag persists across app restarts until the user explicitly starts season 2 — it's a blocking state that prevents task display and cycle check-ins.

- Detection: `newProgramDay === 90 && oldProgramDay < 90` → `seasonComplete: true`
- `useProgramAdvancement` hook sets `isSeasonComplete` state
- Season-complete is a blocking state: no task display, no cycle check-ins
- App layout checks `isSeasonComplete` and redirects to celebration screen
- Flag persists across sessions until `startNewSeason()` is called

## 1.3 Season Report Generation

When the user reaches the season-complete screen, a comprehensive season report is generated by the coaching engine (`generateSeasonReport()`). The report is a narrative story-format summary (compared to the shorter, structured weekly reports) covering: (a) before/after FORGE score comparison, (b) per-pillar movement across the entire 90-day season, (c) habit highlights — which tasks the user completed most consistently, (d) biggest breakthrough pillar, (e) focus area for season 2. The report is generated via Codex AI (Claude proxy edge function) and stored in `weekly_reports` with `week_number = -1` (sentinel value for season report, distinguishing it from weekly reports). If report generation fails, a fallback display shows the raw before/after scores without AI narrative. The report is displayed on the `season-complete.tsx` screen with a celebration animation (confetti or particle effect) before the user transitions to season 2.

- Generated by `generateSeasonReport()` in `coachingEngine.ts` via Claude AI
- Narrative story format: 200–350 words, story arc structure
- Covers: before/after scores, pillar movement, habit highlights, biggest breakthrough, season 2 focus
- Stored in `weekly_reports` table with `week_number = -1` (season report sentinel)
- Falls back to raw score display if AI generation fails (no blank screen)
- Displayed with celebration animation on `season-complete.tsx` screen
- User can review and optionally share the report as a text-based share card

## 1.4 Season Rollover (Cron: Daily 00:00 UTC, season++, program_day=1, Voice Unlock)

Season rollover has two paths: manual (user-initiated from season-complete screen) and cron-triggered (automatic for users who haven't opened the app at day 90).

**Manual rollover**: The `startNewSeason(userId)` function in `programAdvancement.ts` increments `users.season` by 1, resets `program_day` to 1, sets `plan_start_date` to today, and returns `{ newSeason, planGenerated: false }`. The `useProgramAdvancement` hook updates Zustand stores and clears the season-complete blocking state.

**Cron-triggered rollover**: The `season-rollover-handler` edge function runs daily at 00:00 UTC via `pg_cron`. It:
1. Authenticates with `CRON_SECRET` (Authorization: Bearer).
2. Fetches all onboarded users with `plan_start_date` not null.
3. For each user, calculates `programDay` from `plan_start_date` and determines `currentSeason = floor((programDay - 1) / 90) + 1`.
4. If `currentSeason >= 2` and user is still on season 1, rolls them over: updates `users.season = 2`, `progress.program_day = 1`, logs a `season_events` row with event type `season_rollover` and metadata (face_shape, voice_weight_boost, previous_program_day, rollover_date), and resets `daily_tasks.is_completed` for the new season.
5. If `programDay < 90`, marks status as `not_ready` (no action).
6. If user is already past season 2, marks as `skipped`.

**Voice unlock**: Starting in season 2, the voice pillar receives a weight boost based on face shape. The boost values are: oval +0.06, square +0.07, round +0.04, long +0.06, heart +0.05, diamond +0.05. Default is 0.05. This boost is logged in the `season_events` record.

- Manual: `startNewSeason(userId)` — increments season, resets program day to 1, sets new plan_start_date
- Cron: `season-rollover-handler` edge function, triggered daily at 00:00 UTC by `pg_cron`
- Cron auth: `CRON_SECRET` env var, validated via `Authorization: Bearer` header
- Cron logic: calculates program day from start date, determines season number
- Rollover actions: `users.season++`, `progress.program_day = 1`, `season_events` audit log inserted, `daily_tasks.is_completed` reset
- Voice weight boost by face shape: oval 0.06, square 0.07, round 0.04, long 0.06, heart 0.05, diamond 0.05, default 0.05
- Cron returns JSON: `{ rolledOver, totalChecked, results[] }` — each result has `{ userId, status, previousDay?, voiceWeightBoost?, faceShape?, error? }`
- Skips users already on season >= 2; skips users with program day < 90

## 1.5 Season Events Audit Log

The `season_events` table serves as an immutable audit log for all season lifecycle transitions. Each row records: `user_id` (FK to users), `event_type` (currently only `"season_rollover"`), `from_season`, `to_season`, and `event_data` (JSONB with metadata specific to the event). The rollover handler inserts a row with: `face_shape`, `voice_weight_boost`, `previous_program_day`, and `rollover_date`. The table is write-only from the application's perspective — no app screens query it directly. It exists for operational monitoring, debugging rollover failures, and future analytics (e.g., "how many users reached season 2 vs churned at day 90").

- Table: `season_events` with columns `user_id`, `event_type`, `from_season`, `to_season`, `event_data` (JSONB), `created_at` (auto)
- Current event type: `"season_rollover"` only — extensible for future events
- `event_data` JSONB stores: `face_shape`, `voice_weight_boost`, `previous_program_day`, `rollover_date`
- Write-only from app perspective — operational/debugging use only
- Inserted by both manual (`startNewSeason`) and cron (edge function) rollover paths
- Indexed on `user_id` and `created_at` for efficient querying

## 1.6 Phase Transition UI (Day 28/56 Milestone Markers)

The three program phases — Foundation (days 1–30), Activation (days 31–60), Optimisation (days 61–90) — are marked by UI transitions that signal the user's progression through the program. Phase boundaries use day 28 (end of Foundation) and day 56 (end of Activation) as milestone markers with 2-day transition windows.

**Phase transition behaviours**:
- **Day 28**: Category labels transition from generic dots to named categories. The `whyItWorks` text on task details expands by default (previously collapsed). Coaching language shifts from outcome-focused to habit-focused (driven by `stageForDay()`).
- **Day 56**: Task library unlocks advanced-tier tasks for new plan generations. Category labels visible. Coaching language shifts from habit-focused to mechanism-level explanations.
- **Phase name in UI**: Always read from `PHASE_DISPLAY` in `src/constants/phases.ts` — never hardcoded strings.

**Visual markers**: The progress screen and home screen display the current phase name and a phase progress indicator (e.g., "Foundation — Day 12 of 30"). Phase transitions are subtle — no disruptive modals, just UI element unlocks and language shifts. The user should feel natural progression, not abrupt changes.

- Foundation → Activation: day 28 transition, category names appear, whyItWorks default-expand, outcome→habit language
- Activation → Optimisation: day 56 transition, advanced tasks unlock, habit→mechanism language
- Phase names from `PHASE_DISPLAY` constant — never hardcoded in JSX
- Phase progress indicator: "Foundation — Day 12 of 30" format
- Language stage routing via `stageForDay(day)` in `src/lib/languageStage.ts`
- Transition markers should be subtle — no modals or popups that interrupt the user
