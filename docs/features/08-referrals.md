# Feature: Referral System

> Users can share a unique referral code. Track who referred whom via the `users` table (`referral_code`, `referred_by` columns).

Feature: Referral System. Actor: User (Member). Phase: MVP.
Implementation: `supabase/migrations/011_referral_codes.sql`, `src/hooks/useShareCard.ts`
Purpose: Users can share a unique referral code. Track who referred whom via the `users` table (`referral_code`, `referred_by` columns).

---

## 1.1 Code Generation (Auto on User Insert)

Every user receives a unique 6-character alphanumeric referral code automatically when their `users` row is created. The code is generated by a PostgreSQL function `generate_referral_code()` that produces codes from the character set `ABCDEFGHJKLMNPQRSTUVWXYZ23456789` (ambiguous characters `0`, `O`, `1`, `I` excluded). The function loops until a unique code is found — it checks `EXISTS(SELECT 1 FROM users WHERE referral_code = code)` and regenerates if there's a collision. A trigger `trg_users_referral_code` fires `BEFORE INSERT ON users` and sets the code if it's null. The column has a `UNIQUE` constraint and an index (`idx_users_referral_code`) for fast lookups. Existing users without codes were backfilled at migration time. The `referred_by` column is a nullable foreign key referencing `users(id)` with `ON DELETE SET NULL` — if the referrer deletes their account, the referral link is nullified but the referred user's record is preserved.

- 6-character codes from set `ABCDEFGHJKLMNPQRSTUVWXYZ23456789` (34 characters, ambiguous excluded)
- `generate_referral_code()` PL/pgSQL function: loops until unique code found (collision check via `SELECT EXISTS`)
- `trg_users_referral_code` trigger fires `BEFORE INSERT` on `users`, sets `referral_code` if null
- `UNIQUE` constraint on `referral_code` column
- `idx_users_referral_code` index for O(1) lookups by code
- `referred_by` column: `TEXT REFERENCES users(id) ON DELETE SET NULL` (nullable foreign key)
- Backfill: `UPDATE users SET referral_code = generate_referral_code() WHERE referral_code IS NULL` ran at migration time
- Code generation is deterministic and side-effect free — no external API calls, no AI involvement

## 1.2 Code Sharing

Referral codes are surfaced in two sharing flows. The `shareChallenge()` function in `src/services/share/shareService.ts` includes the referral code in the share text as `forge.app/r/{referralCode}`. The `shareStreakMilestone()` function similarly includes the code in its share message. The share card screen (`app/(app)/share-card.tsx`) reads the referral code from `useUserStore` and renders it into the card image. The `ShareScoreCard` component includes the referral code at the bottom of the generated card. Users can also copy their code directly from the profile screen.

- Referral code included in share challenge text: `forge.app/r/{referralCode}`
- Referral code included in streak milestone share text: `forge.app/r/{referralCode}`
- ShareScoreCard component renders referral code in the card footer
- Profile screen displays the user's referral code with a copy-to-clipboard action
- `useShareCard` hook reads `user.referral_code` from `useUserStore`
- Share service functions accept `referralCode` as a parameter, read from store at call time

## 1.3 Referral Tracking

When a new user signs up, they can provide a referral code (passed via deep link or manually entered). The `referred_by` column on their `users` row is populated with the referring user's ID. The database schema supports tracking the referral chain: `referred_by` is a self-referencing foreign key to `users.id`. At MVP, there is no in-app UI showing who a user referred or how many — this data is available via database query and is planned for Phase 2 as a leaderboard/dashboard feature. The `ON DELETE SET NULL` cascade on the foreign key ensures that if the referring user deletes their account, the referred user's record is not deleted — only the `referred_by` field is nullified.

- `referred_by` column on `users` populated at signup when referral code is provided
- Self-referencing FK: `referred_by TEXT REFERENCES users(id) ON DELETE SET NULL`
- Referral chain preserved even if referrer deletes account (SET NULL, not CASCADE)
- MVP: no in-app referral dashboard or referrer stats — data available for DB query
- Phase 2: referral leaderboard showing referred count per user
- Phase 2: "referred by friend" welcome message on signup screen (user story M-65)
