# Feature: AI Coaching Engine

**Actor:** User (Member)
**Phase:** MVP ✅ Done
**Related flows:** [daily-routine-flow.md](../flows/daily-routine-flow.md), [season-complete-flow.md](../flows/season-complete-flow.md)
**Implementation:** `src/services/ai/coachingEngine.ts`, `src/services/ai/prompts/coachingPrompt.ts`, `src/services/ai/prompts/compliPrompt.ts`, `src/lib/languageStage.ts`, `src/hooks/useInsights.ts`, `src/hooks/useWeeklyReport.ts`, `supabase/functions/weekly-report-generator/`

## Purpose

Provide personalised, stage-appropriate coaching that evolves from outcome-focused ("this will make your skin clearer") to mechanism-focused ("retinoids increase cell turnover by binding RAR receptors") as the user progresses through their 90-day season. Coaching is delivered at three cadences: daily insights, weekly reports, and season-end reports.

## System Prompts

All coaching prompts enforce these rules:
- Scientific, specific voice. No exclamation marks.
- Never say: handsome, attractive, hot, ugly, great job.
- Never address the user by name.
- Never use generic motivation ("keep it up", "you're doing amazing", "consistency is key").
- Output exactly the requested format. No preamble, no markdown wraps for single-sentence insights.
- Be specific: reference actual pillars, scores, and tasks — not generic platitudes.

## Subsections

### 5.1 Language Stages

**Implementation:** `src/lib/languageStage.ts`

Coaching vocabulary evolves across three stages as the user gains knowledge and experience. The stage determines how Claude frames insights, reports, and feedback.

**Stage Resolution:**
```typescript
export const stageForDay = (day: number): LanguageStage =>
  day <= 14 ? "outcome" : day <= 30 ? "habit" : "mechanism"
```

| Stage | Days | Vocabulary | Focus | Example |
|-------|------|-----------|-------|---------|
| **Outcome** | 1–14 | Benefit-oriented | "What this does for you" | "Exfoliating twice weekly clears dead skin cells that dull your complexion." |
| **Habit** | 15–30 | Routine-oriented | "Building the system" | "Morning grooming rituals create a feedback loop — each completion makes the next easier." |
| **Mechanism** | 31–90 | Science-oriented | "How it works biologically" | "Retinoids bind to RAR and RXR nuclear receptors, accelerating epidermal turnover by up to 40%." |

**Stage Effects on UI:**
- `showCategoryDots(day)` → days 15-28: pill-shaped category indicators instead of full category names.
- `showCategoryNames(day)` → day 29+: full category names visible.
- `showWhyByDefault(day)` → day 29+: "why it works" section expanded by default on task cards.
- These functions are used by `TaskCard`, `AITipCard`, and coaching prompt builders to tailor display.

**Acceptance Criteria:**
- AC-1.1: `stageForDay(0-14)` returns `"outcome"`.
- AC-1.2: `stageForDay(15-30)` returns `"habit"`.
- AC-1.3: `stageForDay(31-90)` returns `"mechanism"`.
- AC-1.4: UI elements respond to stage correctly (dots, names, why-it-works).
- AC-1.5: Coaching prompts adapt vocabulary to current stage.

### 5.2 Daily Insight

**Implementation:** `coachingEngine.generateDailyInsight(userId, programDay, yesterdayCompletionRate, pillarDeltas, options)`

One sentence summary displayed on the home screen, generated by Claude and cached for 1 hour.

**Generation:**
- Called on home screen mount via `useInsights` hook.
- Two modes:
  - **"daily"**: Standard home screen insight. Prompt: "Generate one daily insight sentence (≤25 words)."
  - **"completion"**: Post-task-completion insight. Prompt: "Generate one sentence referencing today's biggest-leverage task with the outcome it drives (≤25 words)."
- Inputs to Claude:
  - `programDay` — current program day.
  - `yesterdayCompletionRate` — 0-100% completion rate.
  - `pillarDeltas` — current deltas for each pillar (e.g., "skin: +3, grooming: -1, posture: +2").
  - `stage` — from `stageForDay(day)`, passed to `buildCoachingSystemPrompt(stage)`.
- **Constraint:** Output must be ≤ 25 words. Enforced by the system prompt.
- **Fallback:** If Claude call fails, returns: "Complete your daily tasks to move your Optimisation Score."

**Caching:**
- In-memory cache (`insightCache` Map) with key format: `{date}:{stage}:{mode}`.
- TTL: 1 hour (`CACHE_TTL_MS = 60 * 60 * 1000`).
- Cache prevents duplicate Claude calls when user navigates between tabs.
- Cache is memory-only — cleared on app restart.
- `insightDateRef` tracks the date the insight was generated for. If date hasn't changed, reuses cached insight.

**Display:**
- `AITipCard` component on home screen (below task list).
- Shows insight text with a "Coach" label and teal left border accent.
- "Refresh" icon button to force regeneration (bypasses cache).
- Subtitle: "Daily insight — Day {programDay}" in secondary text colour.

**Acceptance Criteria:**
- AC-2.1: Daily insight generated on home screen load.
- AC-2.2: Insight is ≤ 25 words.
- AC-2.3: Insight uses stage-appropriate vocabulary.
- AC-2.4: Cache prevents duplicate generation within 1 hour.
- AC-2.5: Fallback message shown on Claude failure.
- AC-2.6: Refresh button forces regeneration.
- AC-2.7: "Completion" mode triggered after task done.
- AC-2.8: Home screen can render without insight (insight is not blocking).

### 5.3 Weekly Report

**Implementation:** `coachingEngine.generateWeeklyReport(userId, weekNumber, season, pillarDeltas, completionRate)` + edge function `weekly-report-generator`

A 180-250 word markdown coaching report generated weekly for each active user. Combines data analysis with AI-generated narrative.

**Structure:**
1. **Opening observation** (1-2 sentences): Overall performance summary for the week. References completion rate and general trajectory.
2. **Pillar movement breakdown** (2-3 sentences): Highlights key pillars that moved (positive and negative). References actual score changes.
3. **Coaching note on one weakness** (2-3 sentences): Focused analysis of the weakest-performing pillar. Language adapted to stage:
   - **Outcome stage:** "Your posture score dropped slightly this week. Better neck alignment will make your jaw appear sharper — start with the 60-second wall stand."
   - **Habit stage:** "Grooming consistency dipped this week. Consider anchoring your evening routine to an existing habit like brushing your teeth."
   - **Mechanism stage:** "The slight drop in your skin score correlates with reduced UV protection application. UVA rays penetrate glass and degrade collagen by activating MMP-1 enzymes."
4. **Focus for next week** (1-2 sentences): Actionable recommendation with specific task or habit focus.

**Generation Flow:**
1. Edge function `weekly-report-generator` runs every Sunday at 06:00 UTC (via `pg_cron`).
2. Queries all users with `onboarded = true`.
3. For each user:
   - Checks if report already exists for current `week_number` (prevents duplicates).
   - Fetches pillar deltas for the past 7 days from `cycles` table (latest cycle data).
   - Computes completion rate: `completed_count / total_count` for past 7 days from `daily_tasks`.
   - Calls `coachingEngine.generateWeeklyReport()`.
   - Inserts result into `weekly_reports` table.
4. Report stored as `{ weekNumber, season, summaryMd, pillarDeltas }` in `content` column (jsonb).

**Display:**
- Available on `app/(app)/weekly-report/[week].tsx` screen (navigated from home screen or profile).
- Rendered as markdown with proper typography: headings, paragraphs, bold for scores.
- "All Reports" history list with week numbers and date ranges.
- Most recent report preview card on home screen when available.

**Edge Function Security:**
- Authenticated via `cron_secret` stored in Supabase Vault.
- Cron jobs use `verify_cron_secret()` before executing.
- Idempotent: checks `weekly_reports` before generation — safe to re-run.

**Acceptance Criteria:**
- AC-3.1: Weekly report is 180-250 words.
- AC-3.2: Report contains all 4 structural sections.
- AC-3.3: Pillar movement references actual score changes.
- AC-3.4: Coaching note uses stage-appropriate language.
- AC-3.5: Report stored in `weekly_reports` table.
- AC-3.6: Edge function runs every Sunday 06:00 UTC.
- AC-3.7: Duplicate check prevents re-generation for same week.
- AC-3.8: Cron secret authentication prevents unauthorised access.
- AC-3.9: Report renders as markdown on weekly-report screen.
- AC-3.10: Historical reports accessible via list.

### 5.4 Season Report

**Implementation:** `coachingEngine.generateSeasonReport(userId, season, beforeScores, afterScores, biggestMover, habitsStuck, whatChanged, whatDidnt)`

Narrative story-formatted summary generated at the end of each 90-day season. Compares before/after scores, identifies patterns, and sets focus for the next season.

**Structure (Narrative Format):**
- **Opening:** Story-style framing of the 90-day journey. "90 days ago, your skin score sat at 52. Today it's at 68 — a 16-point shift driven by consistent exfoliation and UV protection."
- **Before/After Scores:** All 9 pillars listed with start/end scores and deltas. Biggest mover highlighted.
- **Biggest Mover Deep Dive:** 2-3 sentences on the pillar that improved most, referencing specific habits that drove the change.
- **What Changed:** Habits that were successfully adopted and maintained.
- **What Didn't:** Areas that showed minimal movement — framed factually, not judgmentally.
- **Next Season Focus:** 1-2 sentence recommendation for Season 2 emphasis. References the weakest pillar and suggests new focus areas.
- **Voice:** Scientific, specific, narrative. Always uses mechanism stage language (day 90).

**Inputs:**
- `beforeScores`: PillarScores at season start (from baseline or season rollover snapshot).
- `afterScores`: PillarScores at season end (latest cycle).
- `biggestMover`: Pillar key with largest positive delta.
- `habitsStuck`: Array of habit descriptions that user maintained.
- `whatChanged`: Array of improvement descriptions.
- `whatDidnt`: Array of non-improvement descriptions.

**Generation:**
- System prompt via `SEASON_REPORT_PROMPT` from `compliPrompt.ts`.
- Full narrative style using mechanism stage vocabulary.
- Generated on-demand (not cron-scheduled) when user reaches day 90.
- Displayed on `app/(app)/season-complete.tsx` screen.

**Acceptance Criteria:**
- AC-4.1: Season report generated at day 90.
- AC-4.2: Report uses narrative story format.
- AC-4.3: Before/after scores include all assessed pillars.
- AC-4.4: Biggest mover identified and analysed.
- AC-4.5: Habits that stuck and didn't stick are factual.
- AC-4.6: Next season focus is actionable and specific.
- AC-4.7: Report uses mechanism stage language.
- AC-4.8: Generated on-demand (not scheduled).

### 5.5 Onboarding Compliment

**Implementation:** `coachingEngine.generateOnboardingCompliment(baselineAnalysis)` via `COMPLIMENT_PROMPT` in `compliPrompt.ts`

A 15-25 word specific strength callout based on baseline photo analysis. Displayed on the plan-reveal screen immediately after onboarding — serves as a positive reinforcement that the AI "sees" the user accurately.

**Generation:**
- Called from `useOnboarding.generateCompliment(baselineAnalysis)` after baseline photo is analysed.
- System prompt (`COMPLIMENT_PROMPT`): "You are FORGE's onboarding insight engine. A man just took his baseline photo. Analyse the CycleAnalysis data and identify ONE specific thing he's already doing well — something structural, not flattery."
- Rules:
  - 15-25 words only.
  - Must reference an actual data point from the analysis (a specific pillar score, face shape characteristic, or photo quality aspect).
  - Never say: handsome, attractive, hot, great, amazing.
  - Tone: scientific observation, not motivational speech.
- Example: "Your facial structure shows strong symmetry — the jawline foundation is already well-defined."

**Storage:**
- Stored on `users.initial_compliment` column.
- Displayed once on plan-reveal screen, then accessible from profile screen.
- Fallback if Claude fails: "Your baseline analysis is complete — let's start building on those foundations."

**Acceptance Criteria:**

### 5.6 Edge Function — weekly-report-generator

**Implementation:** `supabase/functions/weekly-report-generator/`

Deno edge function scheduled via `pg_cron` to generate weekly coaching reports for all active users.

**Scheduling:**
- `pg_cron` schedule: `0 6 * * 0` (Sunday, 06:00 UTC).
- Set up via migration `007_schedule_edge_functions.sql`.
- Verified by `cron_secret` stored in Supabase Vault.

**Execution Flow:**
1. Authenticate: `verify_cron_secret(cron_secret)`.
2. Query all users with `onboarded = true`.
3. For each user (sequential to respect rate limits):
   a. Calculate current `week_number` from `plan_start_date`: `ceil(daysSinceStart / 7)`.
   b. Check if report already exists: `SELECT * FROM weekly_reports WHERE user_id = X AND week_number = Y AND season = Z`.
   c. If exists: skip (idempotent).
   d. If not: fetch pillar deltas from most recent `cycles` row.
   e. Compute completion rate: count completed tasks / total tasks for days in this week from `daily_tasks`.
   f. Call `coachingEngine.generateWeeklyReport(userId, weekNumber, season, pillarDeltas, completionRate)`.
   g. This function calls Claude via `claude-proxy` edge function (server-to-server, no browser header needed).
   h. Insert result into `weekly_reports` table: `{ user_id, week_number, season, content: { weekNumber, season, summaryMd, pillarDeltas }, created_at }`.
4. Log statistics: total users processed, reports generated, errors encountered.
5. Return `{ processed: N, generated: N, errors: N }`.

**Error Handling:**
- Individual user failures don't block other users.
- Claude rate limit: if hit, wait and retry with backoff (3 attempts, 5s/15s/45s).
- Supabase errors: logged and skipped — retried on next cron run.
- Cron job timeout: 5 minutes (Deno edge function default). Large user bases may need batching in future.

**Rate Limiting:**
- `claude-proxy` enforces 20 requests per hour per user.
- Weekly reports use the edge function's own user identity (system-level), not individual user tokens — this avoids per-user rate limits for scheduled reports.
- Rate limit table `claude_api_calls` still tracks usage for monitoring.

**Acceptance Criteria:**
- AC-6.1: Edge function runs every Sunday at 06:00 UTC.
- AC-6.2: Authenticated via `cron_secret`.
- AC-6.3: Reports generated for all onboarded users without existing report for current week.
- AC-6.4: Duplicate check prevents re-generation.
- AC-6.5: Individual user failures don't block batch.
- AC-6.6: Rate limit retry with exponential backoff.
- AC-6.7: Report inserted into `weekly_reports` table with correct structure.
- AC-6.8: Function is idempotent — safe to re-run.
