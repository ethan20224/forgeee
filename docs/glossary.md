# Glossary

Definitions for every domain-specific term used across the FORGE system.

---

## Roles

- **User / Member**: A paying subscriber with an active FORGE account. Has a personalised 90-day Season, daily tasks, and progress tracking. Distinct from guest (not yet signed up) and visitor (public).

---

## Program Structure

- **Season**: A 90-day transformation program. Every user starts at Season 1. After day 90, the system automatically rolls over to Season 2 with recalibrated pillar weights, reset program day, and new task library selections. The `season` column on `users` tracks the current season number.

- **Program Day**: The user's current day within a 90-day Season, ranging from 1 to 90. Advances each day the user completes all their tasks. Tracked via `users.program_day` and `progress.last_active_date`. Recovery logic handles lapsed users: if they skip days, they resume at their last program day without penalty to the day count.

- **Phase**: A 30-day segment within a Season. Three phases exist:
  - **Foundation** (days 1–30): outcome language, habit formation, category dots visible from day 15
  - **Activation** (days 31–60): habit language, category names visible, why-it-works expanded by default
  - **Optimisation** (days 61–90): mechanism language, advanced tier tasks unlocked
  Phase names in any user-facing string must come from `PHASE_DISPLAY` in `src/constants/phases.ts`.

- **Week**: A 7-day block within the generated plan. The plan is structured as `weeks[].days[].tasks[]`. Each week has a `weekNumber` assigned during plan generation.

- **Day**: A single program day containing 2–5 tasks. Each day has a `dayNumber` (1–90) and is part of a week. Days are pre-generated by Codex AI during plan generation and persisted to `daily_tasks`.

- **Task**: A single activity the user must complete on a given day. Each task links to a `libraryTaskId` from the Master Task Library and includes a `title`, `pillar` (which of the 9 pillars it affects), `tier` (beginner/intermediate/advanced), `xpValue` (10 base), `duration_mins`, and `whyItWorks` explanation.

- **Master Task Library**: A curated set of ~150 tasks in `src/constants/taskLibrary.ts`. Each task has: `id`, `title`, `pillar`, `tier`, `whyItWorks`, `whyShort`, `outcomeTitle`. Codex selects from this library during plan generation — it never invents new tasks.

- **Outcome Title**: A user-facing alternative title for a task that describes the benefit rather than the activity. Used from day 1–56 (`stageForDay` returns "outcome" or "habit"). Example: "Clearer skin within 30 days" instead of "Morning cleanser routine". Stored in `taskLibrary.outcomeTitle`.

- **Micro Sprint**: A 7-day event-driven override of the standard daily tasks. Triggered by events like a cycle check-in showing a stalled pillar or a user choice. Stored in the `micro_sprints` table with a `sprint_type` and `tasks` JSONB payload. Managed by `src/services/sprints/microSprintEngine.ts`.

## Scoring System

- **Pillar**: One of 9 measurable aspects of appearance and wellness. Each pillar has a score from 0–100 tracked in both `progress` and `cycles` tables:
  1. **Facial Composition** (`facial_composition_score`)
  2. **Skin** (`skin_score`)
  3. **Grooming** (`grooming_score`)
  4. **Hair** (`hair_score`)
  5. **Posture** (`posture_score`)
  6. **Style** (`style_score`)
  7. **Sleep** (`sleep_score`)
  8. **Nutrition** (`nutrition_score`)
  9. **Voice** (`voice_score`)

- **Pillar Score**: A 0–100 value representing the user's current standing in a given pillar. Can change through: (a) cycle photo check-in analysis by Codex Vision, (b) +0.5 drift per task completion via `applyTaskEffect()`, (c) season rollover recalibration.

- **Pillar Weights**: Floating-point values summing to 1.0 that determine each pillar's contribution to the FORGE Optimisation Score. Derived from quiz answers by `deriveInitialWeights()` and adjusted by face shape deltas from `FACE_SHAPE_WEIGHTS`. Stored in `plans.raw_json.pillarWeights`.

- **FORGE Optimisation Score**: A weighted composite 0–100 score calculated as `sum(pillar_score * pillar_weight)` across all 9 pillars. This is the core metric — displayed prominently as the user's primary progress indicator. Computed in `src/services/scores/scoreCalculator.ts`.

- **Delta**: The change in a pillar score since the previous measurement. Crucial context for Codex coaching: `"skin: +3"` means improvement, `"sleep: -2"` means decline.

- **Baseline Photo**: The user's first photo, required for meaningful before/after comparison. Originally captured during onboarding, now deferred to day 3 (prompted in-app via `app/(app)/baseline-photo.tsx`). Analyzed by Codex Vision to establish initial pillar scores and detect face shape. Stored in the private photo bucket with `baseline_photo_path` on `users`.

- **Cycle**: A photo check-in that occurs every 3 days. The user takes front/left/right photos (or picks from gallery), Codex Vision analyzes them, producing a `CycleAnalysis` with per-pillar scores, face shape, overall forge score, primary insight, and next focus area. Results stored in the `cycles` table. `cycle_number` auto-increments per user via trigger.

- **Cycle Analysis**: The structured JSON result from Codex Vision photo analysis. Contains: `cycle_number`, `date`, `photo_quality_flag`, `pillar_scores` (9 objects with `score`, `delta`, `confidence`), `forge_score` (current and cycle_high), `face_shape`, `primary_insight`, `next_focus`.

- **Scan Mode**: Determines which photo angles Codex analyzes. Two modes:
  - `face`: front-facing photo only — excludes style, sleep, nutrition, voice pillars (set to data_gap/null)
  - `full`: front + left + right photos — all pillars analyzed
  The scan mode filtering happens in `applyScanModeFilters()` in `photoAnalyser.ts`.

- **Face Shape**: Detected by Codex Vision from baseline/cycle photos. One of: `oval`, `round`, `square`, `heart`, `diamond`, `triangle`, `oblong`. Stored on `users.face_shape`. Used to apply per-pillar weight adjustments from `faceShapeWeights.ts` when generating plans.

## Progress & Gamification

- **XP (Experience Points)**: Awarded on task completion. Base award is 10 XP per task. Streak bonus of +20 XP applies when `current_streak > 3`. Tracked in `progress.total_xp`. Handled by `src/services/xp/xpService.ts`.

- **Level**: Derived from total XP using the thresholds in `src/constants/levels.ts`. 6 levels total:
  - Level 1: Initiate (0 XP)
  - Level 2: Apprentice (100 XP)
  - Level 3: Adept (300 XP)
  - Level 4: Expert (600 XP)
  - Level 5: Master (1000 XP)
  - Level 6: Grandmaster (1500 XP)

- **Level Name**: Human-readable label for each level (Initiate, Apprentice, Adept, Expert, Master, Grandmaster). Computed by `calculateLevel()` in `xpService.ts`.

- **Streak**: Consecutive days with at least one task completed. Tracked as `current_streak` and `longest_streak` in the `progress` table. Updated each time a task is completed via `updateStreak()` in `streakService.ts`. Milestone detection occurs at days 7, 30, 60, and 90.

- **Streak at Risk**: A state where the user hasn't completed tasks in 2+ days. Previously triggered notifications ("your streak is at risk") which were removed in Phase 9 per the guilt-free notification policy. The streak count itself resets to 0 naturally when a day is completely missed.

- **First Strike**: A Day 1 special module — the user's first-ever task completion. Triggers a celebration modal (`FirstStrikeModal`) with a micro-score and AI-generated insight. Tracked via `users.first_strike_completed`. Handled by `firstStrikeService.ts`.

- **Heatmap**: A 91-cell grid (days 1–90 + today) showing task completion status per day. Each cell has a status: `done` (all tasks completed that day), `partial` (some completed), `missed` (day passed, no tasks done), or `future` (day not yet reached).

## Coaching & AI

- **Daily Insight**: One sentence (≤25 words) generated by Codex each day. Based on yesterday's completion rate, current pillar deltas, and the current program day's language stage. Displayed on the home screen via `AITipCard`. Handled by `generateDailyInsight()` in `coachingEngine.ts`.

- **Weekly Report**: A 180–250 word markdown report generated every Sunday. Includes pillar movement breakdown, coaching note, and next week's focus. Stored in `weekly_reports` table as JSONB. AI generated by the `weekly-report-generator` edge function (cron: Sunday 06:00 UTC). Users see it in-app via `app/(app)/weekly-report/[week].tsx`.

- **Season Report**: End-of-Season narrative report in story format. Compares before/after scores, identifies the biggest mover, calls out habits that stuck, and sets focus for the next season. Uses MECHANISM language stage (day 90 by definition). Handled by `generateSeasonReport()` in `coachingEngine.ts`.

- **Language Stage**: Determines the vocabulary Codex uses in coaching output. Three stages determined by `stageForDay(day)`:
  - **Outcome** (days ≤14): "This will make your skin clearer" — benefits-focused
  - **Habit** (days 15–30): "Keep doing this and it becomes automatic" — consistency-focused
  - **Mechanism** (days 31+): "Retinoids increase cell turnover by binding RAR receptors" — biology-focused

- **Onboarding Compliment**: A 15–25 word strength callout generated after baseline photo analysis. Displayed on the plan-reveal screen. Highlights the user's strongest pillar from the baseline analysis.

- **Rate Limit**: Codex API calls are capped at 20 per user per hour via the `claude-proxy` edge function. Tracks calls in the `claude_api_calls` audit table.

## Subscription

- **Subscription Tier**: One of `none`, `premium`, or `max`. Currently only `premium` is active — the **Basic** tier at $11/mo. Tracked on `users.subscription_tier`. Synced by RevenueCat SDK and the `revenuecat-webhook` edge function.

- **RevenueCat**: The in-app purchase SDK used for subscription management. Handles purchase, restore, and status queries. Communicates with Supabase via the `revenuecat-webhook` edge function for server-side validation of `INITIAL_PURCHASE`, `RENEWAL`, `CANCELLATION`, and other events.

---

## System States

- **Onboarded**: Boolean on `users` tracking whether the user has completed onboarding. Set to `true` only after plan is saved to database AND `plan_start_date` is set. The app router guards on this flag: `onboarded=false` redirects to auth screens, `onboarded=true` redirects to the main app.

- **Auth Checked**: Session state flag in `userStore. isAuthChecked`. Prevents flash-of-wrong-screen during cold start while the Supabase session is being recovered from SecureStore/AsyncStorage.

- **AI Simulation Mode**: Development toggle controlled by `EXPO_PUBLIC_AI_SIMULATION` env var. When enabled, `isAiSimulationEnabled()` returns `true` and mock data is returned instead of real Codex calls. Saves API costs during development and testing. Config in `src/config/aiSimulation.ts`.

- **Dev Bypass**: RevenueCat development mode detected by `isDevBypass()` in `revenuecat.ts`. Returns mock subscription data when `__DEV__` is true and `APP_ENV !== "production"`. Allows testing the full app flow without real purchases.

- **Pending Task Effects**: A queue (`pending_task_effects` table) of score drifts (+0.5) that should be applied when a task is completed. If `applyTaskEffect()` fails (e.g. network error), the effect is queued and retried on next app launch via `retryPendingEffects()`.

## Data Flow

- **Plan Cache**: Module-level in-memory storage (`src/lib/planCache.ts`) used to hand off the generated plan between `plan-loading` → `plan-reveal` screens during onboarding. Lives for the app session only. Also caches baseline analysis and estimated score.

- **Quiz Cache**: Module-level in-memory storage (`src/lib/quizCache.ts`) for quiz answers during the multi-step quiz flow. Also persists drafts to AsyncStorage so the user can resume the quiz after closing the app.

- **Signed URL**: A time-limited URL (default 3600 seconds) generated for private bucket photos. Since photos are in a private bucket (`user-photos-private`), the app generates signed URLs via `getSignedPhotoUrl()` for rendering and sending to Codex Vision. Managed by `src/services/image/signedUrl.ts`.

- **Session Recovery**: On cold start, the root layout (`app/_layout.tsx`) subscribes to `supabase.auth.onAuthStateChange`. If no session is seen, it falls back to `supabase.auth.getSession()` which reads directly from storage. A 1-second retry is attempted if the first DB fetch fails.
